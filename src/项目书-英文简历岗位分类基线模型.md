# 基于英文简历的岗位类别分类项目书

## 一、项目名称

基于英文简历文本的岗位类别自动分类系统

## 二、项目背景

在招聘场景中，企业每天会收到大量求职者简历。人工筛选简历不仅耗时，而且容易受到主观因素影响。若能够根据简历内容自动判断候选人所属岗位方向，例如人力资源、信息技术、教师、销售、金融、工程等类别，就可以辅助 HR 快速完成简历初筛、岗位归类和后续分发。

本项目计划先使用原版英文简历数据集构建基线模型，验证简历文本分类任务的可行性。基线阶段采用两条路线：第一条是 `TF-IDF + XGBoost`，用于建立结构清晰、可解释性较强的机器学习基线；第二条是 `FastText supervised`，用于建立轻量级深度学习文本分类基线。

后续可在 XGBoost 和 FastText 的结果基础上升级为 BERT 系列模型。由于简历文本普遍较长，普通 BERT 直接截断会丢失后半部分经历、技能和教育信息，因此本项目在训练 BERT 时使用 `bert-base-uncased` 作为底层编码器，并在其外部自定义长文本分块、chunk 聚合和分类层。本文档中的 `Hierarchical BERT` 指“Base BERT 编码器 + chunk 级编码 + 文档级聚合”的长文本分类结构，不是直接加载现成的 Hierarchical BERT 预训练模型。

## 三、数据集介绍

### 1. 数据集来源

该数据集来自 LiveCareer 网站中的简历示例，由数据集作者通过网页爬虫采集整理。

数据集中包含 2400 多份英文简历，每份简历被标注为一个岗位类别。原始数据同时包含文本格式和 HTML 格式，页面介绍中也说明 PDF 文件按照类别文件夹进行存储，每个 PDF 文件名与 CSV 中的 `ID` 对应。

### 2. 数据集用途

该数据集用于将给定简历自动分类到指定岗位类别中，是一个典型的多分类文本分类任务。

输入：

```text
英文简历文本
```

输出：

```text
岗位类别
```

例如：

```text
输入：包含招聘、员工关系、薪酬福利、培训等内容的英文简历
输出：HR
```

```text
输入：包含 Python、SQL、Linux、系统开发、数据库等内容的英文简历
输出：INFORMATION-TECHNOLOGY
```


暂不使用 `Resume_html` 和 PDF 文件，避免增加解析复杂度。

当前已构建的数据集划分如下：

| 数据文件 | 样本数量 | 用途 |
| --- | ---: | --- |
| `train_data.txt` | 2034 | 模型训练 |
| `test_data.txt` | 230 | 最终测试评估 |

当前项目暂不单独设置验证集，训练集和测试集比例约为 9:1。训练阶段不使用测试集选择最优模型，测试集只在训练结束后用于最终评估，避免测试集信息泄漏。

文本长度统计如下：

| 指标 | 数值 |
| --- | ---: |
| 平均长度 | 约 6295 字符 |
| 中位数 | 约 5886 字符 |
| 最短文本 | 21 字符 |
| 最长文本 | 38842 字符 |

原始统计中最短文本仅 21 字符，这类样本通常缺少完整经历、技能和教育信息，已判定为噪声风险样本。当前训练和测试文件中已经不存在该类极短样本，`train_data.txt` 最短样本为 789 字符，`test_data.txt` 最短样本为 683 字符。为避免后续重新构建数据时再次引入类似噪声，代码中新增过短样本过滤规则：

```text
min_text_chars = 200
min_total_tokens = 50
```

简历文本整体偏长，因此后续若使用 BERT，需要考虑最大长度截断、长文本分块或摘要策略。

针对当前 `train_data.txt` 和 `test_data.txt` 进一步统计 token 长度分布后发现，极少数超长简历会明显拉高最大长度，不适合直接按照最长样本设置窗口。由于当前环境统计时无法调用 HuggingFace tokenizer，本次统计使用正则 token 近似计数，结果如下：

| 数据集 | 平均 token 数 | 中位数 | 最大值 | P95 | P98 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 训练集 | 984 | 915 | 6188 | 1681 | 2101 |
| 测试集 | 951 | 906.5 | 3277 | 1704 | 1990 |

因此，Hierarchical BERT 阶段不再按照最长样本设置窗口，而是采用覆盖约 97% 到 98% 样本的固定窗口：

```text
max_chunks = 4
pad_size = 512
max_tokens_per_chunk = 510
max_total_tokens = 2040
```

超过 2040 个正文 token 的极端长样本会在数据加载阶段被过滤，不直接送入模型。按该阈值估算：

| 数据集 | 原始样本数 | 保留样本数 | 过滤样本数 | 保留比例 |
| --- | ---: | ---: | ---: | ---: |
| 训练集 | 2034 | 1986 | 48 | 97.64% |
| 测试集 | 230 | 225 | 5 | 97.83% |

这样可以避免极端长样本决定模型窗口，降低显存压力和训练时间，同时保留绝大多数有效样本。

当前训练集、测试集和完整 `train.txt` 在 `min_text_chars = 200`、`min_total_tokens = 50` 的过短样本规则下，均不会额外删除样本；该规则主要用于防止后续数据重构时引入 21 字符这类噪声文本。

## 八、基线模型建设目标

本阶段目标不是直接追求最优模型，而是先构建一个可运行、可评估、可复现的 baseline。

主要目标包括：

1. 完成原始英文简历数据读取和清洗。
2. 完成岗位类别标签编码。
3. 使用 TF-IDF 提取文本特征。
4. 训练 XGBoost 和 FastText 两类 baseline 模型。
5. 输出准确率、宏平均 F1、加权 F1、分类报告和混淆矩阵。
6. 保存 XGBoost 模型、FastText 模型、TF-IDF 向量化器和标签映射文件。
7. 为后续 BERT 模型升级提供对比基准。

## 九、技术路线

### 1. 数据读取

读取原始 CSV：

```python
import pandas as pd

df = pd.read_csv("Resume.csv")
```

使用字段：

```python
texts = df["Resume_str"]
labels = df["Category"]
```

### 2. 数据清洗

基线阶段建议进行轻量清洗：

- 去除首尾空格。
- 合并多余空白字符。
- 去除异常过短文本。
- 保留英文大小写或统一小写。
- 暂不删除专业词汇。

示例：

```python
import re

def clean_text(text):
    text = str(text)
    text = re.sub(r"\s+", " ", text)
    return text.strip()

df["clean_text"] = df["Resume_str"].apply(clean_text)
```

### 3. 标签编码

将岗位类别转换为数字标签：

```python
from sklearn.preprocessing import LabelEncoder

label_encoder = LabelEncoder()
df["label"] = label_encoder.fit_transform(df["Category"])
```

保存 `label_encoder`，方便预测结果还原为类别名称。

### 4. 数据集划分

由于类别不均衡，建议使用分层抽样。当前项目只划分训练集和测试集，不单独设置验证集，比例约为 9:1：

```python
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    df["clean_text"],
    df["label"],
    test_size=0.1,
    random_state=42,
    stratify=df["label"]
)
```

训练阶段仅使用训练集进行参数更新；测试集不参与调参和最优模型选择，只在训练完成后进行最终评估。

### 5. XGBoost 特征工程

XGBoost 不能直接处理原始文本，因此需要先使用 TF-IDF 将英文简历文本转换为数值向量。

建议初始参数：

```python
from sklearn.feature_extraction.text import TfidfVectorizer

vectorizer = TfidfVectorizer(
    lowercase=True,
    stop_words="english",
    ngram_range=(1, 2),
    max_features=50000,
    min_df=2,
    max_df=0.9
)
```

说明：

| 参数 | 作用 |
| --- | --- |
| `lowercase=True` | 统一转小写 |
| `stop_words="english"` | 去除英文停用词 |
| `ngram_range=(1, 2)` | 使用 unigram 和 bigram |
| `max_features=50000` | 限制最大特征数量 |
| `min_df=2` | 过滤只出现 1 次的低频词 |
| `max_df=0.9` | 过滤过高频词 |

如果 TF-IDF 特征维度过高，XGBoost 训练速度较慢，可以考虑两种优化方式：

- 降低 `max_features`，例如设置为 20000 或 30000。
- 使用 `TruncatedSVD` 对 TF-IDF 特征降维，再输入 XGBoost。

### 6. FastText 数据格式

FastText 是轻量级文本分类模型，适合快速训练英文文本分类任务。FastText supervised 模式要求训练数据格式如下：

```text
__label__类别 文本内容
```

例如：

```text
__label__HR HR specialist with experience in recruiting employee relations onboarding benefits and training.
__label__INFORMATION-TECHNOLOGY Software engineer with Python SQL Linux database development and cloud service experience.
```

因此需要将原始数据转换为 FastText 训练文件：

```text
fasttext_train.txt
fasttext_test.txt
```

转换逻辑：

```python
def to_fasttext_line(text, label):
    text = clean_text(text)
    return f"__label__{label} {text}"
```

### 7. 基线模型选择

本项目基线阶段确定使用两个模型：

| 模型 | 输入特征 | 定位 |
| --- | --- | --- |
| XGBoost | TF-IDF 向量 | 机器学习强基线 |
| FastText | 原始英文文本 | 轻量级深度学习文本分类基线 |

两种模型的作用不同：

| 模型 | 优点 | 可能问题 |
| --- | --- | --- |
| XGBoost | 对结构化特征和非线性关系建模能力强，可作为机器学习强基线 | TF-IDF 特征维度高时训练较慢 |
| FastText | 训练速度快，适合文本分类，能利用词和 n-gram 信息 | 语义理解能力弱于 BERT |

推荐 baseline 对比方式：

```text
XGBoost(TF-IDF) vs FastText(supervised)
```

如果后续 BERT 模型上线，则最终形成三组对比：

```text
XGBoost baseline
FastText baseline
BERT model
```

## 十、模型评估方案

由于该任务是 20 类多分类，即使删除了样本最少的类别，仍需要关注各类别上的表现，因此不能只看 Accuracy。

建议评估指标：

| 指标 | 说明 |
| --- | --- |
| Accuracy | 整体预测正确率 |
| Macro Precision | 各类别精确率平均，关注小类别 |
| Macro Recall | 各类别召回率平均，关注小类别 |
| Macro F1 | 各类别 F1 平均，适合不均衡数据 |
| Weighted F1 | 按类别样本数加权后的 F1 |
| Confusion Matrix | 查看类别之间混淆情况 |

重点关注：

```text
Macro F1
```

因为多分类任务中不同岗位类别的文本相似度较高，Accuracy 可能掩盖部分类别识别效果差的问题。

当前训练代码已加入 dev 集，因此模型训练和评估流程调整为：

```text
训练集：用于模型参数学习
dev 集：用于 early stopping 和最优模型选择
测试集：仅在训练结束后进行最终评估
```

训练过程中使用 dev 集进行 early stopping 和最优模型选择，不使用测试集进行早停、调参或模型选择。模型训练完成后加载 dev Macro F1 最优权重，再在测试集上输出最终 Accuracy、Precision、Recall、F1、分类报告和混淆矩阵。这样可以减少测试集信息泄漏，使最终测试结果更接近真实泛化表现。

## 十三、风险与问题

### 1. 数据量较小

删除 4 个低样本类别后，正式建模数据量为 2264 条，共 20 个类别，平均每类约 113 条样本。虽然类别分布更加均衡，但整体数据量仍然不算大。

解决思路：

- baseline 阶段先使用删除低样本类别后的 20 类数据训练。
- 后续如模型效果不足，可继续进行数据增强。
- 使用分层抽样，保证训练集和测试集中的类别比例一致。
- 当前暂不设置验证集，避免在数据量较小的情况下进一步压缩训练样本。

### 2. 类别不均衡

原始数据中 `BPO`、`AUTOMOBILE`、`AGRICULTURE`、`DIGITAL-MEDIA` 样本偏少，已在正式建模前删除。删除后剩余类别分布更加接近，但仍需关注各类别的 Precision、Recall 和 F1。

解决思路：

- 评估时重点看 Macro F1。
- 使用分层抽样划分训练集和测试集。
- 如果仍有类别效果明显偏低，再考虑定向数据增强。

### 3. 文本较长

简历平均长度超过 6000 字符，BERT 无法一次性读取完整文本。同时，少量极端长样本会显著拉高最大长度，如果直接按照最长样本设置窗口，会导致训练显存占用过高、速度变慢。

解决思路：

- XGBoost 阶段使用 TF-IDF，不受 512 token 限制影响明显。
- FastText 阶段可直接处理较长文本，但仍建议清理异常空白和极端过长文本。
- Hierarchical BERT 阶段采用固定窗口方案：每份简历最多保留 4 个 chunk，每个 chunk 长度为 512 token。
- 使用 `max_total_tokens = 2040` 作为过长样本过滤阈值，超过该阈值的极端长样本在数据加载阶段过滤。
- 窗口设计依据训练集和测试集 token 分布，而不是依据最长样本，从而避免少量异常长文本影响整体训练效率。

### 4. 数据来源偏单一

数据来自简历示例网站，可能存在模板化、领域偏差和真实简历差异。

解决思路：

- 在项目报告中说明数据来源和局限性。
- 后续可引入更多公开简历或岗位描述数据。

## 十四、后续升级方向

baseline 完成后，可以进一步升级：

### 1. XGBoost 与 FastText 对比分析

基线阶段结束后，首先对 XGBoost 和 FastText 进行对比：

| 对比维度 | XGBoost | FastText |
| --- | --- | --- |
| 输入 | TF-IDF 特征 | 原始文本 |
| 训练速度 | 中等，受特征维度影响 | 较快 |
| 可解释性 | 可通过特征重要性分析 | 相对较弱 |
| 文本语义理解 | 依赖词频和 n-gram | 利用词向量和 n-gram |
| 适合作用 | 机器学习强基线 | 轻量文本分类基线 |

### 2. Hierarchical BERT 长文本分类方案

本项目深度学习主线方案采用 `bert-base-uncased` 作为 base BERT 编码器，并在其外部构建层级分块聚合分类器。为便于表述，下文仍将该结构简称为 Hierarchical BERT。

当前代码实际使用的底层编码器为：

```text
bert-base-uncased
```

输入：

```text
Resume_str
```

输出：

```text
20 类岗位类别
```

核心思想是将一份长简历切分为多个短文本片段，每个片段长度不超过 base BERT 的最大输入长度，再分别输入 `bert-base-uncased` 编码，最后将多个片段的语义表示聚合为整份简历的表示，用于岗位类别分类。

整体流程如下：

```text
Resume_str
-> 文本清洗
-> 按 token 窗口切分为多个 chunk
-> 每个 chunk 输入 bert-base-uncased
-> 得到多个 chunk embedding
-> chunk 表示聚合
-> 分类层
-> 20 类岗位类别
```

示例：

```text
Resume_str
-> chunk_1: tokens 1-512
-> chunk_2: tokens 513-1024
-> chunk_3: tokens 1025-1536
-> chunk_4: tokens 1537-2048
```

每个 chunk 单独经过 base BERT：

```text
chunk_1 -> bert-base-uncased -> CLS_1
chunk_2 -> bert-base-uncased -> CLS_2
chunk_3 -> bert-base-uncased -> CLS_3
chunk_4 -> bert-base-uncased -> CLS_4
```

再进行聚合：

```text
[CLS_1, CLS_2, CLS_3, CLS_4]
-> mean pooling / attention pooling
-> document embedding
-> classifier
```

最终输出：

```text
HR / INFORMATION-TECHNOLOGY / SALES / FINANCE / TEACHER / ...
```

#### 推荐初始参数

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 底层编码器 | `bert-base-uncased` | 代码实际训练使用的 base BERT |
| 每个 chunk 长度 | 512 tokens | 与 BERT 最大输入长度一致 |
| 每份简历最大 chunk 数 | 4 个 | 根据当前长度分布确定，控制显存和训练速度 |
| 每份简历正文 token 上限 | 2040 tokens | 由 `4 × 510` 得到，超过该长度的极端长样本过滤 |
| chunk 聚合方式 | mean pooling | 先用简单稳定方案 |
| 后续升级聚合方式 | attention pooling | 让模型学习不同片段的重要性 |
| 分类类别数 | 20 | 对应删除低样本类别后的 20 个岗位类别 |

当前窗口不是按最长样本设置，而是根据训练集和测试集的 P95/P98 长度分布确定。这样可以覆盖绝大多数样本，同时避免少数 3000 到 6000 token 以上的极端样本显著增加显存消耗。

#### 为什么适合本项目

Hierarchical BERT 适合本项目的原因：

- 简历文本很长，直接截断会丢失信息。
- 简历天然具有层次结构，例如 Summary、Skills、Experience、Education。
- 每个片段可以独立编码，再汇总为整份简历表示。
- 相比 Longformer，Hierarchical BERT 更贴合当前 BERT 升级路线。
- 对当前 2264 条、20 类样本规模更稳，训练资源要求也更可控。
- 通过固定 4 chunk 窗口和过长样本过滤，可以避免极端长简历主导窗口设计。

#### 最终模型路线

本项目模型路线调整为：

| 阶段 | 模型 | 作用 |
| --- | --- | --- |
| 基线模型 1 | XGBoost + TF-IDF | 建立机器学习基线 |
| 基线模型 2 | FastText supervised | 建立轻量文本分类基线 |
| 后续主模型 | Base BERT + 层级分块聚合分类器 | 处理长简历文本，作为最终深度学习模型 |

#### 当前已落地优化点

结合当前代码实现，Hierarchical BERT 阶段已经完成以下优化：

| 优化点 | 当前方案 | 作用 |
| --- | --- | --- |
| 模型结构 | `bert-base-uncased` 作为 chunk 编码器，外接文档级聚合和分类层 | 明确使用 base BERT 训练，避免误用 Common Core 或其他现成层级分类模型 |
| 长文本窗口 | 每份简历最多 4 个 chunk，每个 chunk 512 tokens | 控制显存占用和训练时间 |
| 过短样本处理 | 过滤少于 200 字符或少于 50 token 的样本 | 去除缺少有效信息的噪声文本 |
| 过长样本处理 | 过滤超过 2040 个正文 token 的极端长样本 | 避免最长样本主导窗口设计 |
| 聚合方式 | 使用 `chunk_mask` + mean pooling | 忽略 padding chunk，得到稳定的文档向量 |
| 数据划分 | 训练集内部切分 dev 集，测试集只做最终评估 | 支持 early stopping，避免测试集泄漏 |
| 训练策略 | 使用 dev Macro F1 保存最优模型，训练结束后评估测试集 | 提升泛化稳定性，减少测试集泄漏 |
| 模型保存 | `model/best_hierarchical_bert.pt` | 保存 dev Macro F1 最优权重和类别映射信息 |

### 3. 长文本处理

由于简历文本较长，当前项目采用如下确定方案：

- 先根据标点、简历小标题和 token 数量将简历切分为多个语义相对完整的 chunk。
- 每个 chunk 的最大输入长度为 512 tokens，其中正文 token 上限为 510。
- 每份简历最多保留 4 个 chunk，模型窗口约为 2040 个正文 token。
- 训练集和测试集中超过 2040 个正文 token 的极端长样本在数据加载阶段过滤。
- 对剩余样本，使用 BERT 分别编码每个 chunk，并取 `[CLS]` 向量作为 chunk 表示。
- 使用 `chunk_mask` 忽略 padding 出来的空 chunk，再通过 mean pooling 得到整份简历的向量表示。

该方案的核心原则是：窗口大小由样本长度分布决定，而不是由最长样本决定。

### 4. 数据增强

删除低样本类别后，数据平衡性已经明显改善。后续如果仍需增强，可以优先根据最终测试集分类报告和错误分析结果，对表现较差的类别进行定向扩充，例如 Recall 较低或容易与其他岗位混淆的类别。

扩充目标可以设置为每类 150 条、200 条，或根据阶段性测试结果和人工错误分析动态调整。

## 十七、总结

本项目计划基于英文简历数据集构建岗位类别自动分类系统。当前阶段先使用原始英文数据训练两个 baseline 模型：`TF-IDF + XGBoost` 和 `FastText supervised`。前者作为机器学习强基线，后者作为轻量级深度学习文本分类基线。

这两个 baseline 模型将为后续 BERT 模型提供基础评估参考。最终主线模型使用 `bert-base-uncased` 作为 base BERT 编码器，通过将长简历切分为多个 chunk，再聚合多个 chunk 的语义表示，解决普通 BERT 单次输入长度不足的问题。项目完成后，可以形成一个完整的简历分类流程，为招聘系统中的简历归类、岗位推荐和人才库管理提供基础能力。
